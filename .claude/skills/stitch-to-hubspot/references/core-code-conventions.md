# Core Code Convention Framework

This reference describes the Core Code boilerplate structure and the conventions that all generated modules must follow. The boilerplate provides layout infrastructure, design tokens, macros, and global styles. It does not provide content modules - those are generated by the skill.

## Theme Configuration

### theme.json

```json
{
  "label": "Core Code",
  "preview_path": "./templates/home.html",
  "responsive_breakpoints": [
    {
      "name": "mobile",
      "mediaQuery": "@media (max-width: 767px)",
      "previewWidth": { "value": 477 }
    }
  ],
  "version": "1.0"
}
```

### fields.json (theme-level)

CMS-configurable spacing exposed in Theme Settings:

| Field path | Type | Default | Range |
|---|---|---|---|
| `spacing.mobile.vertical_spacing` | number (slider) | 30 | 0-500 |
| `spacing.mobile.horizontal_spacing` | number (slider) | 24 | 0-500 |
| `spacing.desktop.vertical_spacing` | number (slider) | 80 | 0-500 |
| `spacing.desktop.horizontal_spacing` | number (slider) | 40 | 0-500 |
| `spacing.max_width` | number (slider) | 1440 | 1200-2000, step 10 |

### theme_overrides.css

Bridges theme fields to CSS custom properties:

```css
:root {
  --containerWidth: {{ theme.spacing.max_width ~ 'px' }};
  --moduleTopSpacing: {{ theme.spacing.mobile.vertical_spacing ~ 'px' }};
  --moduleRightSpacing: {{ theme.spacing.mobile.horizontal_spacing ~ 'px' }};
  --moduleBottomSpacing: {{ theme.spacing.mobile.vertical_spacing ~ 'px' }};
  --moduleLeftSpacing: {{ theme.spacing.mobile.horizontal_spacing ~ 'px' }};
}

@media (width >= 992px) {
  :root {
    --moduleTopSpacing: {{ theme.spacing.desktop.vertical_spacing ~ 'px' }};
    --moduleRightSpacing: {{ theme.spacing.desktop.horizontal_spacing ~ 'px' }};
    --moduleBottomSpacing: {{ theme.spacing.desktop.vertical_spacing ~ 'px' }};
    --moduleLeftSpacing: {{ theme.spacing.desktop.horizontal_spacing ~ 'px' }};
  }
}
```

## Design Tokens

All tokens defined in `base.html` inline `<style>` at `:root`. The skill overwrites values per project but preserves the token structure.

### Grid
`--gridGutters` (20px), `--gridRowGap`, `--gridColGap`, `--gridColumns` (12)

### Layout
`--rootFontSize` (16), `--siteHeaderHeight` (80px), `--containerWidth` (1440px)

### Z-index Scale
| Token | Value | Use |
|---|---|---|
| `--zBelow` | -1 | Behind content |
| `--zDefault` | 1 | Default |
| `--zDrawer` | 100 | Drawers, dropdowns |
| `--zSticky` | 200 | Sticky elements |
| `--zHeader` | 300 | Site header |
| `--zOverlay` | 400 | Overlay backgrounds |
| `--zModal` | 500 | Modal dialogs |
| `--zPopover` | 600 | Popovers |
| `--zTooltip` | 700 | Tooltips |

### Spacing Scale (13 steps)
| Token | Value |
|---|---|
| `--space0` | 0 |
| `--space4` | 4px |
| `--space8` | 8px |
| `--space12` | 12px |
| `--space16` | 16px |
| `--space20` | 20px |
| `--space24` | 24px |
| `--space32` | 32px |
| `--space40` | 40px |
| `--space48` | 48px |
| `--space64` | 64px |
| `--space80` | 80px |
| `--space96` | 96px |
| `--space128` | 128px |

Always snap to the nearest step. Never invent intermediate values.

### Border
`--border1` (1px), `--border2` (2px), `--border3` (3px)

### Radius
| Token | Value |
|---|---|
| `--radiusNone` | 0 |
| `--radiusSm` | 4px |
| `--radiusMd` | 8px |
| `--radiusLg` | 16px |
| `--radiusXl` | 32px |
| `--radiusAvatar` | 50% |

Note: the `frontend-standards` design-tokens reference uses a slightly extended radius scale (`--radiusXxs`, `--radiusXs`, `--radiusSm`, `--radiusMd`, `--radiusLg`, `--radiusXl`). When generating modules, use whichever token set is present in the project's `base.html`. If the project has the extended scale, use it. If it has the base scale above, use that. Do not mix scales.

### Colours
**Greys:** `--grey10` through `--grey90`
**Brand base:** `--primary10`, `--primary20`, `--primary30`
**Semantic:** `--primaryColour`, `--secondaryColour`, `--tertiaryColour`, `--highlightColour`
**Text:** `--textColour`, `--textColourLight`, `--textColourInverse`
**State:** `--colorSuccess`, `--colorWarning`, `--colorError`, `--colorInfo`
**Base:** `--white`, `--black`

### Animation
`--transitionSpeed` (0.3s), `--easeIn`, `--easeOut`, `--easeInOut`

### Typography
**Families:** `--fontPrimary` (default: Montserrat), `--fontSecondary` (default: Open Sans)
**Weights:** `--fontLight` (300), `--fontRegular` (400), `--fontMedium` (500), `--fontSemibold` (600), `--fontBold` (700), `--fontXbold` (800)
**Fluid sizes (clamp-based):** `--h1` through `--h6`, `--pXs` through `--pXl`

### Icons
`--iconSm` (52px), `--iconBase` (72px), `--iconLg` (100px)

### Other
`--touchArea` (48px), `--dotDimension` (12px)

### Project-specific tokens

When the design requires tokens not in the base set, add them at the end of the relevant category in the `:root` block. Comment them clearly:

```css
/* ---- Project-specific: Surfaces ---- */
--backgroundColour: #fafaf3;
--surfaceLow: #f4f4ee;
--surfaceMid: #eeeee8;

/* ---- Project-specific: Typography ---- */
--displaySize: clamp(3rem, 5vw + 1rem, 6rem);
--fontBlack: 900;

/* ---- Project-specific: Radius ---- */
--radiusFeature: 3rem;
--radiusCta: 4rem;
```

## Template Hierarchy

All page templates extend `layouts/base.html` and define a `dnd_area`.

### base.html provides

- `<!DOCTYPE html>` and `<html>` with `lang` attribute
- `<head>`: meta tags, critical CSS inline, async font loading, async stylesheet loading (`main.css`, `theme_overrides.css`), Google Fonts
- `<body>`: skip link, header partial, `{% block body %}`, footer partial, async JS loading
- Macro imports (globally available in all templates and modules)

### Partials

- **`partials/header.html`** - Structural shell. Contains the `<header>` element, skip link target, and a slot for the navigation module. The slot mechanism depends on the boilerplate implementation - either a static `{% module %}` tag pointing to the generated navigation module, or a `{% dnd_area %}` allowing the editor to swap navigation modules. Use `{% module %}` by default (simpler, less risk of editors breaking the nav).
- **`partials/footer.html`** - Structural shell. Contains the `<footer>` element and a slot for the footer content module. Same mechanism as header.

The skill generates the navigation and footer content modules that populate these shells. When generating the page template, do not include navigation or footer modules in the main `dnd_area` - they live in the partials.

**Wiring the partial to the generated module:**

In `partials/header.html`, the slot references the generated navigation module by path:
```hubl
<header class="c-header js-header">
  {% module "site_navigation"
    path="../modules/site_navigation"
  %}
</header>
```

In `partials/footer.html`:
```hubl
<footer class="c-footer">
  {% module "footer_content"
    path="../modules/footer_content"
  %}
</footer>
```

The module paths must match the generated module directory names exactly.

### Page templates

Generated per project. Pattern:

```hubl
{% extends "./layouts/base.html" %}

{% block body %}
{% dnd_area "main_content" %}

  {% dnd_section %}
    {% dnd_module
      path="../modules/module_name",
      field_name="default value"
    %}
    {% end_dnd_module %}
  {% end_dnd_section %}

{% end_dnd_area %}
{% endblock body %}
```

Module paths reference the flat `modules/` directory: `path="../modules/hero_banner"`, `path="../modules/service_grid"`, etc.

### System templates

`templates/system/` contains 404, 500, password prompt, email preferences (7 templates). These extend `base.html` but do not use DnD areas. Rarely customised.

## Macro API

Globally imported in `base.html`. Generated modules must use these macros instead of raw HTML for the elements they cover.

| Macro file | Alias | Key exports | Use for |
|---|---|---|---|
| `link_helpers.html` | `link_helper` | `render_link()`, `render_cta()` | All links and CTA buttons |
| `image_helpers.html` | `image_helper` | `render_responsive_images()` | All content images |
| `video_helpers.html` | `video_helper` | Video embed helpers | Video embeds |
| `text_helpers.html` | `text_helpers` | `render_heading(model, enabled, name)` | All module headings |
| `style_helpers.html` | `style_helpers` | `render_toggle()` | Style toggles |

### Deprecated macros - do not use

- `link_helper.render_button()` - cross-macro calls fail silently
- `link_helper.render_module_footer_cta()` - calls render_button internally
- `style_helpers.render_module_padding()` - removed entirely

### Macro usage examples

**Heading:**
```hubl
{% if module.module_heading.heading %}
  {{ text_helpers.render_heading(module.module_heading, module.module_heading.heading, "module_heading") }}
{% endif %}
```

**Link:**
```hubl
{% if module.module_footer_cta.link.url.href %}
  <div class="m-moduleName__footerCta">
    {{ link_helper.render_link(module.module_footer_cta.link) }}
  </div>
{% endif %}
```

**Image:**
```hubl
{% if module.image.src %}
  {{ image_helper.render_responsive_images(module.image) }}
{% endif %}
```

## CSS Architecture

### Entry points

- `css/main.css` - primary stylesheet, loaded async via `rel="preload"`. Uses HubL `{% include %}` to compose sub-files.
- `css/theme_overrides.css` - loaded async, maps theme fields to CSS custom properties.
- Critical CSS inlined in `<head>` via `base.html`: `layout.critical.css`, `header.critical.css`, `typography.critical.css`.
- Module CSS auto-loaded by HubSpot when the module is placed on a page.

### Global CSS files (provided by boilerplate)

| Directory | Files | Purpose |
|---|---|---|
| `css/critical/` | layout, header, typography | Inlined in `<head>`, above-fold rendering |
| `css/global/elements/` | buttons, forms, tables, typography, images, skip-link | Base HTML element styles |
| `css/global/objects/` | containers_dnd, page | Layout objects and DnD wrapper handling |
| `css/global/utilities/` | Utility classes | Minimal utility layer |

### Naming convention

- BEM with camelCase: `.m-featureCards__heading--active`
- ITCSS prefixes: `.o-` (objects), `.c-` (components), `.e-` (elements), `.m-` (modules)
- CSS variables: camelCase (`--primaryColour`, `--space20`)
- State classes: `is-active`, `is-hidden`, `is-loading`, `has-error`
- JS hook classes: `js-moduleName-role`

### Module spacing pattern

Modules inject per-instance spacing via inline `style` on `.o-wrapper--module`:

```html
<div class="o-wrapper o-wrapper--module" style="
  --moduleTopSpacingMobile: {{ module.style.spacing_alignment.spacing.mobile.padding_top }}px;
  --moduleBottomSpacingMobile: {{ module.style.spacing_alignment.spacing.mobile.padding_bottom }}px;
  --moduleTopSpacingDesktop: {{ module.style.spacing_alignment.spacing.desktop.padding_top }}px;
  --moduleBottomSpacingDesktop: {{ module.style.spacing_alignment.spacing.desktop.padding_bottom }}px;">
```

The global CSS for `.o-wrapper--module` reads these custom properties to apply responsive padding.

### Breakpoints

- Primary: 992px (desktop)
- Secondary: 768px (tablet)
- Range syntax only: `@media (width >= 992px)`
- Mobile-first: base styles apply at all sizes. Media queries add desktop overrides.

## Module Generation Patterns

### Directory structure

All modules live in a flat `modules/` directory:

```
modules/
├── hero_banner.module/
│   ├── meta.json
│   ├── fields.json
│   ├── module.html
│   ├── module.css
│   └── module.js (if interactive)
├── service_grid.module/
├── site_navigation.module/
└── footer_content.module/
```

Directory names use `lowercase_underscores.module`.

**Naming uniqueness:** If a design has two structurally different sections serving a similar purpose (e.g. two distinct testimonial layouts), give each a unique name reflecting its specific role: `testimonial_quote.module` and `testimonial_carousel.module`, not `testimonial_1.module` and `testimonial_2.module`. If a design reuses the same module pattern multiple times with different content (e.g. three text-and-media sections), generate one module and place it three times in the template with different field defaults.

### Field ordering in fields.json

HubSpot renders fields in the editor sidebar in the order they appear in the JSON array. Follow this ordering:

1. Content-specific fields and groups (the module's unique fields - what the editor interacts with most)
2. `module_heading` group
3. `module_footer_cta` group
4. `scrollid`
5. `style` group (on the STYLE tab - appears separately in the editor regardless of position, but keep it last for file readability)

### Section background colours

Module background colours are handled by the module itself, not by the `dnd_section` wrapper. Each module that needs a background colour variant should include a background colour choice field in its `style` group as a sibling of `spacing_alignment`:

```json
{
  "name": "background_colour",
  "label": "Background Colour",
  "type": "choice",
  "choices": [
    ["default", "Default"],
    ["surface", "Surface"],
    ["primary", "Primary"],
    ["secondary", "Secondary"]
  ],
  "default": "default"
}
```

The module's HubL applies the selected background via a class modifier, and the module's CSS maps each option to the appropriate token. This keeps the module self-contained and avoids relying on DnD section styling which content editors may inadvertently override.

**HubL (in module.html on the outer wrapper):**
```hubl
<div class="o-wrapper o-wrapper--module m-serviceGrid--bg-{{ module.style.background_colour }}" style="...">
```

**CSS (in module.css):**
```css
.m-serviceGrid--bg-default {
  background-color: transparent;
}

.m-serviceGrid--bg-surface {
  background-color: var(--surfaceLow);
}

.m-serviceGrid--bg-primary {
  background-color: var(--primaryColour);
  color: var(--textColourInverse);
}

.m-serviceGrid--bg-secondary {
  background-color: var(--secondaryColour);
  color: var(--textColourInverse);
}
```

When a dark background is selected, ensure child text elements inherit the inverse colour or are explicitly set. Check contrast ratios for all combinations.

### meta.json

```json

  "css_assets": [],
  "js_assets": [],
  "categories": ["DESIGN"],
  "icon": "module",
  "is_available_for_new_content": true
}
```

### module.html skeleton

```hubl
<div class="o-wrapper o-wrapper--module" style="
  --moduleTopSpacingMobile: {{ module.style.spacing_alignment.spacing.mobile.padding_top }}px;
  --moduleBottomSpacingMobile: {{ module.style.spacing_alignment.spacing.mobile.padding_bottom }}px;
  --moduleTopSpacingDesktop: {{ module.style.spacing_alignment.spacing.desktop.padding_top }}px;
  --moduleBottomSpacingDesktop: {{ module.style.spacing_alignment.spacing.desktop.padding_bottom }}px;"
  {% if module.scrollid %}id="{{ module.scrollid }}"{% endif %}>

  <div class="m-heroBanner o-container">

    {# Module heading #}
    {% if module.module_heading.heading %}
      {{ text_helpers.render_heading(module.module_heading, module.module_heading.heading, "module_heading") }}

      </div>
```

## Shared Field Groups

These JSON structures are included in every content module's `fields.json` unless explicitly inappropriate for the module.

### Module heading group (CONTENT tab)

```json
{
  "name": "module_heading",
  "label": "Module Heading",
  "type": "group",
  "children": [
    {
      "name": "heading",
      "label": "Heading",
      "type": "text",
      "default": ""
    },
    {
      "name": "heading_type",
      "label": "Heading Level",
      "type": "choice",
      "choices": [
        ["h2", "H2"],
        ["h3", "H3"],
        ["h4", "H4"]
      ],
      "default": "h2"
    },
    {
      "name": "subheading",
      "label": "Subheading",
      "type": "text",
      "default": ""
    }
  ]
}
```

### Module footer CTA group (CONTENT tab)

```json
{
  "name": "module_footer_cta",
  "label": "Footer CTA",
  "type": "group",
  "children": [
    {
      "name": "link",
      "label": "Link",
      "type": "link",
      "default": {
        "url": { "href": "" },
        "open_in_new_tab": false
      }
    }
  ]
}
```

### Style group (STYLE tab)

```json
{
  "name": "style",
  "label": "Style",
  "type": "group",
  "tab": "STYLE",
  "children": [
    {
      "name": "spacing_alignment",
      "label": "Spacing",
      "type": "group",
      "children": [
        {
          "name": "spacing",
          "label": "Spacing",
          "type": "spacing",
          "default": {
            "mobile": { "padding_top": 40, "padding_bottom": 40 },
            "desktop": { "padding_top": 80, "padding_bottom": 80 }
          }
        }
      ]
    }
  ]
}
```

### Scroll ID (CONTENT tab)

```json
{
  "name": "scrollid",
  "label": "Scroll ID",
  "type": "text",
  "default": ""
}
```

### When to omit shared groups

- **Navigation modules:** Omit `module_heading`, `module_footer_cta`, and `scrollid`. Navigation has its own field structure.
- **Footer content modules:** Omit `module_heading` if the footer design has no visible heading. Keep `style` for spacing control.
- **Utility modules** (e.g. a bare rich text block): Omit `module_heading` and `module_footer_cta`. Keep `style` and `scrollid`.

## JavaScript Conventions

See the `frontend-standards` JavaScript reference for the full standard. Key points for generated modules:

- Vanilla JS only. IIFE with `'use strict'`.
- `const`/`let` only. Never `var`.
- `.js-*` classes for DOM targeting. `data-*` for configuration.
- Event delegation on the module container.
- `CoreCode.trapFocus()` and `CoreCode.debounce()` from shared utilities.
- `window.lazyModuleInit(moduleId, initFn)` for below-fold modules. This defers `initFn` until the module element (identified by `moduleId`) enters the viewport via IntersectionObserver. Defined in `js/modules/utilities.js`.
- Guard against missing elements. Graceful degradation if JS fails.

See `references/interactive-module-guardrails.md` for pattern-specific JS requirements.

## Vendor Libraries

Available in `js/vendor/`. Load on demand - only include when the module requires them.

| Library | Purpose | When to use |
|---|---|---|
| Splide | Carousels, sliders | Complex multi-slide carousels. For simple single-slide, generate vanilla JS instead. |
| Plyr | Video player | Custom video player UI beyond native controls. |
| lite-youtube | YouTube embed | Lightweight YouTube embed (performance). |
| Odometer | Animated counters | Number counting animations. Consider vanilla JS for simple cases. |
| Tabs Accordion | Tabs/accordion | Available but consider generating vanilla JS for better control over ARIA and keyboard patterns. |

When using a vendor library, the interactive module guardrails still apply. Configure the library with accessible defaults and add any missing keyboard/ARIA support.
