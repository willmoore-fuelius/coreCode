> Learnings from the `feature/landingPage` branch development (6 modules built: hero_landing, cta_banner, steps, icon_cards, service_cards, reviews, accordion). These should be fed back into the Core Code boilerplate to prevent recurring issues on future projects.

## 1. Critical Bug: `link_helpers.html` Macro Internal Calls Silently Fail

**Severity:** Critical ‚Äî causes invisible rendering failures with no error output.

**Problem:** The macros in `macros/link_helpers.html` (`render_module_footer_cta`, `render_button`, `render_link`, `render_cta`) call each other internally **without namespace prefixes**. When imported via `{% import ... as link_helper %}` in `base.html`, these internal calls silently produce empty output ‚Äî no HubL error is thrown.

**Failure chain:**
```
render_module_footer_cta(button, name)
  ‚Üí render_button(button, scope)        ‚Üê no namespace prefix
      ‚Üí render_link(link_settings, ...) ‚Üê no namespace prefix ‚Äî SILENTLY FAILS
```

**What you see:** The wrapper `<div class="m-moduleFooter">` renders, but it's completely empty inside. Debug output confirms all data is correct ‚Äî the macro just produces nothing.

**Recommended fix for boilerplate:**

Option A ‚Äî **Refactor macros to use self-referencing namespace** (if HubL supports it):
```hubl
{# Inside link_helpers.html, call sibling macros with explicit namespace #}
{% macro render_button(button, scope, classes) %}
  {# Use fully-qualified call instead of bare render_link() #}
  {{ link_helper.render_link(link_settings, ...) }}
{% endmacro %}
```

Option B ‚Äî **Flatten the call chain** so each macro is self-contained and doesn't call sibling macros internally.

Option C ‚Äî **Document the workaround** and provide an inline rendering snippet in the module scaffolder docs (current approach ‚Äî works but is more verbose).

**Inline workaround pattern (proven in production):**
```html
{%- set footer = module.module_footer_cta -%}
{%- if footer.button_type == 'link' and footer.link_settings.link_field.url.href and footer.link_settings.link_text -%}
<div class="m-moduleFooter">
  <a id="{{ name }}__footer_cta"
     class="e-button e-button--{{ footer.button_style|default('primary') }}"
     href="{{ footer.link_settings.link_field.url.href }}"
     {% if footer.link_settings.link_field.open_in_new_tab %}target="_blank" rel="noopener" aria-label="{{ footer.link_settings.link_text }} (opens in new window)"{% else %}aria-label="{{ footer.link_settings.link_text }}"{% endif %}>
    {{ footer.link_settings.link_text }}
  </a>
</div>
{%- elif footer.button_type == 'cta' and footer.cta_settings.cta_field -%}
<div class="m-moduleFooter">
  {% cta guid="{{ footer.cta_settings.cta_field }}" %}
</div>
{%- endif -%}
```

## 2. `meta.json` ‚Äî Missing Keys Cause Cryptic Errors

**Problem:** Several upload errors trace back to incomplete `meta.json` files, but error messages are unhelpful (`"internal error"`, `"CSS or Javascript is not allowed"`).

**Recommended boilerplate `meta.json` template:**
```json
{
  "label": "Module Name",
  "description": "Brief description of what the module does",
  "icon": "../images/module_icons/module_name.svg",
  "css_assets": [],
  "js_assets": [],
  "host_template_types": ["PAGE", "BLOG_POST", "BLOG_LISTING"],
  "categories": ["BODY_CONTENT"],
  "smart_type": "NOT_SMART",
  "is_available_for_new_content": true
}
```

**Key rules to document:**
| Rule | Why |
|------|-----|
| `host_template_types` is **mandatory** when `css_assets` or `js_assets` exist | Without it, HubSpot blocks all CSS/JS |
| `categories` must be **UPPERCASE** | `"body_content"` ‚Üí error; `"BODY_CONTENT"` ‚Üí works |
| Never include `content_tags` | Auto-generated by HubSpot; including it causes deserialization errors |
| Never include `module_id` | Auto-generated on first upload |
| `description` should always be present | Prevents vague `"internal error"` on some deployments |

## 3. `fields.json` ‚Äî Group-Level Defaults Corrupt Field Data

**Problem:** Adding a `"default"` object at the group level (alongside `"children"` with their own defaults) can corrupt HubSpot's field data resolution. Fields appear in the editor but return unexpected values or empty data at render time.

**Wrong:**
```json
{
  "name": "card_settings",
  "type": "group",
  "default": { "title": "Hello" },
  "children": [
    { "name": "title", "type": "text", "default": "Hello" }
  ]
}
```

**Correct ‚Äî defaults only on individual fields:**
```json
{
  "name": "card_settings",
  "type": "group",
  "children": [
    { "name": "title", "type": "text", "default": "Hello" }
  ]
}
```

**Exception:** Repeater groups use `"default": []` (an array) at the group level to define the initial set of items ‚Äî this is fine.

## 4. Repeater Fields ‚Äî Use `group_occurrence_meta` (Not `occurrence`)

**Problem:** Early development used `"occurrence"` for repeater config. The correct property is `"group_occurrence_meta"` with `min`, `max`, `sorting_label_field`, and `default` count.

**Correct pattern:**
```json
{
  "name": "items",
  "type": "group",
  "group_occurrence_meta": {
    "min": 0,
    "max": null,
    "sorting_label_field": "heading",
    "default": 3
  },
  "default": [
    { "heading": "Item 1" },
    { "heading": "Item 2" },
    { "heading": "Item 3" }
  ],
  "children": [ ... ]
}
```

**Note:** The `"default"` at group level here is an **array** (not an object) ‚Äî this is the correct way to define initial repeater items.

## 5. Spacing Group ‚Äî Strict Placement, Parameter Path, and Field Names

**Problem:** The spacing macro call is sensitive to both placement and the exact object path passed to it. Additionally, the spacing child field names must match what the macro reads.

**Rules for boilerplate:**
1. Spacing group must be the **last group** in `fields.json`
2. The macro call goes **outside** the module's root `<div>`/`<section>`:
   ```html
   </section>
   {%- if module.scrollid -%}</div>{%- endif -%}
   {{ style_helpers.render_module_padding(name, module.style.spacing_alignment.spacing) }}
   ```
3. The parameter path is exactly `module.style.spacing_alignment.spacing` ‚Äî not `module.style` or `module.style.spacing_alignment`
4. **Child field names must be `padding_top` and `padding_bottom`** ‚Äî NOT `top`/`bottom`. The macro reads `spacing.mobile.padding_top`, `spacing.mobile.padding_bottom`, `spacing.desktop.padding_top`, `spacing.desktop.padding_bottom`. Using `top`/`bottom` silently produces no padding (renders empty string).
   ```json
   { "name": "padding_top", "label": "Top", "type": "number", "default": 80 },
   { "name": "padding_bottom", "label": "Bottom", "type": "number", "default": 80 }
   ```
5. Standard defaults: mobile 80/80, desktop 120/120 (except hero/full-bleed which use 0/0)
6. **Do not use `render_module_padding` macro ‚Äî inline the `<style>` block directly in `module.html`.** HubSpot suppresses or deduplicates dynamic `<style>` output from macros, causing CMS spacing controls to have no effect. The reliable pattern is a bare `<style>` tag directly in the module template, scoped by `#{{ name }}`:
   ```html
   <style>
     #{{ name }} .o-wrapper--module {
       --moduleTopSpacing: {{ module.style.spacing_alignment.spacing.mobile.padding_top }}px;
       --moduleBottomSpacing: {{ module.style.spacing_alignment.spacing.mobile.padding_bottom }}px;
     }
     @media (min-width: 992px) {
       #{{ name }} .o-wrapper--module {
         --moduleTopSpacing: {{ module.style.spacing_alignment.spacing.desktop.padding_top }}px;
         --moduleBottomSpacing: {{ module.style.spacing_alignment.spacing.desktop.padding_bottom }}px;
       }
     }
   </style>
   ```

## 6. Macro Imports ‚Äî No Imports in `module.html`

**Problem:** New developers may try to add `{% import %}` statements in module templates. All macros are already imported globally in `base.html`.

**Rule:** Never add macro import statements to `module.html`. The following aliases are available everywhere:
- `link_helper` (singular) ‚Äî from `Link Helpers.html`
- `text_helpers` (plural) ‚Äî from `Text Helpers.html`
- `image_helpers` (plural) ‚Äî from `Image Helpers.html`
- `style_helpers` (plural) ‚Äî from `Style Helpers.html`
- `video_helper` (singular) ‚Äî from `Video Helpers.html`

**Watch out:** HubSpot converts filenames to Title Case on deployment (`link_helpers.html` ‚Üí `Link Helpers.html`). The `base.html` imports use Title Case paths. And note the inconsistent singular/plural aliases ‚Äî `link_helper` and `video_helper` are singular; the rest are plural.

## 7. Image Field Defaults ‚Äî All Keys Required

**Problem:** Incomplete image field defaults cause rendering issues or editor errors.

**Complete image field default:**
```json
{
  "type": "image",
  "default": {
    "size_type": "auto",
    "src": "https://placehold.co/1200x800",
    "alt": "Descriptive alt text",
    "loading": "lazy",
    "width": 1200,
    "height": 800
  }
}
```

All six keys (`size_type`, `src`, `alt`, `loading`, `width`, `height`) should be present.

## 8. Link Field Defaults ‚Äî All Keys Required

**Problem:** Missing keys in link field defaults can trigger `'link' is required but no default is set` errors, sometimes cascading from unrelated meta.json issues.

**Complete link field default:**
```json
{
  "type": "link",
  "default": {
    "url": {
      "type": "EXTERNAL",
      "href": ""
    },
    "open_in_new_tab": false,
    "no_follow": false
  }
}
```

## 9. `main.css` ‚Äî Must Use HubL `{% include %}`, Not CSS `@import`

**Problem:** CSS `@import` doesn't work in HubSpot's asset pipeline. The `main.css` entry point uses HubL template includes to compose the stylesheet.

**Correct pattern in `main.css`:**
```css
/* Generic */
{% include '../global/generic/normalize.css' %}
{% include '../global/generic/reset.css' %}

/* Objects */
{% include '../global/objects/wrapper.css' %}
```

## 10. HubL Variable Scoping ‚Äî Arrays in Loops Don't Persist

**Problem:** Variables (especially arrays) modified inside `{% for %}` loops don't persist outside the loop in HubL due to scoping rules.

**Workaround:** Use `{% set %}` with namespace objects or build output inline within the loop.

## 12. HubSpot Upload Error Quick Reference

Add this table to the boilerplate README or developer docs:

| Error Message | Root Cause | Fix |
|---------------|-----------|-----|
| `internal error` | Missing `description`, `icon`, or other required meta.json keys | Add all required meta.json fields |
| `CSS or Javascript is not allowed on modules with ANY content type` | Missing `host_template_types` in meta.json | Add `"host_template_types": ["PAGE", "BLOG_POST", "BLOG_LISTING"]` |
| `'link' is required but no default is set` | Incomplete link field default, or cascading from broken meta.json | Fix meta.json first; ensure all link fields have complete defaults |
| `'X' is not a valid category` | Lowercase category value | Use UPPERCASE: `"BODY_CONTENT"`, `"MEDIA"`, `"TEXT"` |
| `Cannot deserialize value of type java.util.ArrayList` | `content_tags` included in meta.json | Remove `content_tags` entirely ‚Äî it's auto-generated |
| `Value for field with an occurrence must be an array` | Repeater default is `{}` instead of `[]` | Use `"default": []` or `"default": [{...}]` |

## 16. Repeater Groups: `default` Array Required to Pre-populate Items

**Severity:** Medium ‚Äî causes confusing "No items yet" state in editor even when `occurrence.default` is set.

**Problem:** Setting `occurrence.default: 3` on a repeater group only tells HubSpot how many blank items to create ‚Äî it does **not** populate those items with field values. Without a `default` array at the group level, all items are created empty and the editor list shows them with no label content.

**Fix:** Always add a `default` array alongside `occurrence` on any repeater group that should have pre-populated content. Each object in the array corresponds to one pre-populated item and must include values for every child field that has content:

```json
{
  "name": "items",
  "label": "Steps",
  "type": "group",
  "occurrence": {
    "min": 1,
    "max": 6,
    "default": 3,
    "sorting_label_field": "items.heading"
  },
  "default": [
    {
      "heading": "Step one heading",
      "body_text": "<p>Body copy here.</p>",
      "image": {
        "src": "https://example.com/image.jpg",
        "alt": ""
      },
      "button": {
        "button_type": "link",
        "button_style": "primary",
        "link_settings": {
          "link_field": { "url": { "type": "EXTERNAL", "href": "https://example.com" }, "open_in_new_tab": false, "no_follow": false },
          "link_text": "Learn more"
        },
        "cta_settings": {}
      }
    }
  ],
  "children": [ ... ]
}
```

**Rule:** Whenever a repeater group has default content (images, headings, copy, buttons), the `default` array at group level must be populated with the same number of items as `occurrence.default`. The child field-level `default` values only apply when a user manually adds a new item ‚Äî they do not seed the initial state.

## 17. Form Field Picker Only Works with Legacy/Classic Forms

**Severity:** Medium ‚Äî causes "No results found" in the form picker with no indication why.

**Problem:** HubSpot's `"type": "form"` field in `fields.json` and the `hbspt.forms.create()` JS API only support **legacy/classic forms**. Forms created using the **new-style form editor** (URL contains `/new-editor/`) do not appear in the module form picker and cannot be embedded via the JS API.

**Symptoms:**
- Form picker search returns "No results found"
- "Select a form" dropdown is empty
- The form exists and is published in the same portal

**Fix:** Always create forms using the **legacy form builder** (Marketing > Forms > choose "Regular form" / classic editor). New-style forms are incompatible with custom module form embeds.

**Note:** This is a HubSpot platform limitation, not a code issue. The `"type": "form"` field definition and `hbspt.forms.create()` embed pattern are correct.
